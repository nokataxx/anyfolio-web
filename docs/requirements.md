# anyfolio 要件定義書

> 自分のナレッジを、どのデバイスからでも美しく閲覧できるアプリケーション

---

## 1. プロジェクト概要

### コンセプト
Obsidian・GdriveなどでMarkdownファイルを管理しているユーザーが、スマホや外出先のPCからでも、認証された安全な環境でナレッジを美しく閲覧できるWebアプリ＋スマホアプリ。

### 背景・課題
- GdriveのMDファイルをスマホで開くとプレーンテキストで表示されるだけ
- Obsidian Syncは有料で敷居が高い
- Markdownを綺麗にレンダリングしてくれるスマホアプリが少ない
- PDFも含めたナレッジを一元管理・閲覧できる場所がない

### ターゲットユーザー
- Obsidianユーザー
- Markdownで情報管理しているフリーランス・エンジニア
- マルチデバイスでナレッジにアクセスしたいユーザー

## 3. 共通要件

### 3-1. 対応ファイル形式

anyfolio は **Markdown中心のナレッジ管理** を基本思想とし、テキスト系ファイルはアップロード時にMarkdownへ変換して保存する。これにより閲覧・編集・検索の体験がすべて `.md` に統一される。

| ファイル形式 | フェーズ |
|------------|---------|
| `.md` (Markdown) | Phase 1 |
| `.pdf` | Phase 1 |
| `.xlsx` / `.xls` (Excel) | Phase 1 |
| `.pptx` / `.ppt` (PowerPoint) | Phase 1（アップロード時にPDFへ変換して保存・閲覧） |
| `.txt` (テキスト) | Phase 1（アップロード時にMarkdownへ変換して保存・閲覧） |
| `.docx` (Word) | Phase 1（アップロード時にMarkdownへ変換して保存・閲覧） |
| `.doc` (Word旧形式) | Phase 1（テキスト抽出後にMarkdownとして保存） |
| 画像（png/jpg/jpeg/gif/webp/svg） | Phase 1 |
| その他ドキュメント | Phase 2以降 |

### 3-2. 認証

- **Supabase Auth** を使用
- メールアドレス＋パスワードによるログイン
- 将来的にGoogle OAuthも対応可能
- 未認証状態でのアクセスはログイン画面にリダイレクト
- セッションはブラウザ・アプリに保持（次回自動ログイン）

### 3-3. Supabaseデータベース設計

#### `anyfolio_folders` テーブル
| カラム | 型 | 説明 |
|-------|-----|------|
| `id` | uuid | PK |
| `user_id` | uuid | 所有者（auth.uid） |
| `name` | text | フォルダ名 |
| `parent_id` | uuid | 親フォルダ（nullはルート） |
| `created_at` | timestamp | 作成日時 |

#### `anyfolio_files` テーブル
| カラム | 型 | 説明 |
|-------|-----|------|
| `id` | uuid | PK |
| `user_id` | uuid | 所有者（auth.uid） |
| `folder_id` | uuid (nullable) | 所属フォルダ（nullはルート直下） |
| `name` | text | ファイル名 |
| `type` | text | `md` / `pdf` / `xlsx` / `pptx` / `image`（pptxはPDF変換後、txt/docx/docはMarkdown変換後のパスを保持） |
| `storage_path` | text | Supabase Storage上のパス |
| `created_at` | timestamp | 作成日時 |
| `updated_at` | timestamp | 更新日時 |

### 3-4. Supabase Storage

- バケット名：`anyfolio-files`
- Row Level Security（RLS）を適用
- 自分のファイルのみアクセス可能（URLを知っていてもアクセス不可）

### 3-5. PowerPoint PDF変換

`.pptx` / `.ppt` ファイルはアップロード時にPDFに変換し、変換後のPDFをStorageに保存する。閲覧時は既存のPDFビューアをそのまま利用する。

#### 変換フロー
```
ユーザーが .pptx をアップロード
        ↓ クライアントサイドで変換処理
        ↓ 変換後のPDFをStorageに保存
        ↓ anyfolio_files に type='pdf' として登録
```

#### 注意事項
- 変換中はローディング表示を行い、完了後に自動で表示を更新する
- 変換エラー時はユーザーに通知する
- `.ppt`（レガシー形式）もサポート

### 3-6. Markdown変換（txt / docx / doc）

`.txt` / `.docx` / `.doc` ファイルはアップロード時にMarkdownへ変換し、変換後の `.md` ファイルをStorageに保存する。保存後は通常のMarkdownファイルとして閲覧・編集・検索が可能。

#### 変換方針

| 入力 | 変換方法 | 備考 |
|------|---------|------|
| `.txt` | 拡張子を `.md` に変更（内容はそのまま） | プレーンテキストはMarkdownとして妥当 |
| `.docx` | **mammoth.js** の `convertToMarkdown` | 見出し・太字・斜体・リスト・リンク・表を保持 |
| `.doc` | テキスト抽出後に `.md` として保存 | 旧バイナリ形式のため構造は失われる |

#### 変換フロー
```
ユーザーが .txt / .docx / .doc をアップロード
        ↓ クライアントサイドでMarkdownに変換
        ↓ 変換後の .md をStorageに保存
        ↓ anyfolio_files に type='md' として登録（name は .md 拡張子）
```

#### 注意事項
- 変換中はローディング表示を行い、完了後に自動で表示を更新する
- 変換エラー時はユーザーに通知する
- 変換後は通常のMDファイルとして扱われ、本アプリのMarkdown編集機能で編集可能

### 3-7. セキュリティ（RLS）

```sql
-- 自分のファイルのみ操作可能
create policy "自分のファイルのみ"
on anyfolio_files for all
using (auth.uid() = user_id);

create policy "自分のフォルダのみ"
on anyfolio_folders for all
using (auth.uid() = user_id);
```

### 4-4. フルテキスト検索

ファイル名検索（サイドバー内）とは独立した検索ダイアログで、ファイルの内容を全文検索する機能。`Cmd+K`（Mac）/ `Ctrl+K`（Windows）のショートカットで起動する。

#### 対応ファイル形式

| ファイル形式 | テキスト抽出方法 |
|------------|----------------|
| `.md` | テキストをそのまま検索（txt/docx/docから変換されたものも含む） |
| `.pdf` | pdfjs-dist でテキスト抽出 |
| `.xlsx` | SheetJS でセル内容をテキスト抽出 |

#### UI仕様

```
┌─────────────────────────────────────────────┐
│ 🔍 ファイル内容を検索...              [ESC] │
├─────────────────────────────────────────────┤
│ 📄 ファイル名.md — フォルダ名               │
│   ...マッチした箇所のプレビュー（前後文脈）...│
│                                             │
│ 📄 ファイル名.pdf — フォルダ名              │
│   ...マッチした箇所のプレビュー（前後文脈）...│
│                                             │
│ （検索結果をクリックすると該当ファイルを表示）│
└─────────────────────────────────────────────┘
```

- モーダルダイアログ形式で画面中央に表示
- 検索結果にはマッチ箇所の前後テキスト（コンテキスト）を表示
- マッチ箇所のキーワードをハイライト表示
- 検索結果クリックで該当ファイルを開く
- `ESC` キーまたはオーバーレイクリックで閉じる

#### 実装フェーズ

| フェーズ | 方式 | 特徴 |
|---------|------|------|
| Phase A（初期実装） | クライアントサイド検索 | 検索時にSupabase Storageからファイルをダウンロードしてテキスト抽出・検索。DB変更不要でシンプル |
| Phase B（拡張） | DB全文検索 | `anyfolio_files` テーブルに `content_text` カラムを追加し、PostgreSQL全文検索（`tsvector`/`tsquery`）を利用。高速・スケーラブル |

##### Phase A: クライアントサイド検索

```
ユーザーが検索キーワードを入力
        ↓ 全ファイルのメタデータを取得（既存の useAllFiles）
        ↓ 対象ファイルをSupabase Storageからダウンロード
        ↓ ファイル形式に応じてテキスト抽出
        ↓ クライアントサイドでテキストマッチング
        ↓ マッチ箇所のプレビュー付きで結果表示
```

- デバウンス（300ms）で入力中の過剰なリクエストを防止
- 抽出済みテキストはメモリ内にキャッシュ（セッション中の再検索を高速化）
- ローディングインジケータ表示

##### Phase B: DB全文検索（将来拡張）

```
ファイルアップロード時
        ↓ テキスト抽出
        ↓ anyfolio_files.content_text に保存
        ↓ PostgreSQL tsvector インデックスで索引化

検索時
        ↓ Supabase RPC でサーバーサイド全文検索
        ↓ マッチ箇所・スニペットをレスポンスで返却
```

**DBスキーマ変更（Phase B）:**

| カラム | 型 | 説明 |
|-------|-----|------|
| `content_text` | text (nullable) | 抽出済みテキスト内容 |
| `content_tsv` | tsvector | 全文検索用インデックス |

**RLS:** 既存ポリシー（`auth.uid() = user_id`）がそのまま適用される。

## 6. Gdrive連携要件

### 基本方針

| フェーズ | 方式 | 内容 |
|---------|------|------|
| Phase 1〜5 | Supabaseアップロード | 手動でファイルをアップロードして管理 |
| Step 6以降 | Gdrive直接参照 | Google Drive APIでファイルを取得・表示 |

### Step 6以降 Gdrive連携の仕様

#### 認証
- Google OAuth 2.0 でGdriveへのアクセス権を取得
- Supabase AuthのGoogle OAuth と統合

#### 動作イメージ
```
ユーザーがGdriveフォルダを指定
        ↓ Google Drive API
  フォルダ内のMD・PDFを取得
        ↓
  anyfolioで綺麗にレンダリング
```

#### 機能
| 機能 | 説明 |
|------|------|
| フォルダ指定 | GdriveのフォルダをanyfolioにマウントするUIで指定 |
| 自動同期 | Gdrive上のファイル変更を自動反映 |
| Obsidian Vault対応 | GdriveにバックアップしたVaultフォルダをそのまま参照 |
| キャッシュ | 一度取得したファイルをSupabaseにキャッシュして高速表示 |

#### メリット
- アップロード作業が不要になる
- Gdriveが唯一のソースとして管理がシンプル
- ObsidianのVaultフォルダをそのままanyfolioで閲覧可能
- スマホからもGdrive上のファイルをリアルタイムで閲覧可能

## 7. Markdown編集機能要件

### 基本方針

閲覧専用アプリから、軽量なナレッジ編集環境へ拡張する。
フル機能エディタ（IDE相当）は目指さず、外出先での手軽な修正・メモ追記に最適化する。

アップロード時の自動Markdown変換（Section 3-6）により、`.txt` / `.docx` / `.doc` 由来のファイルも `.md` として本編集機能の対象になる。

### 機能一覧

| 機能 | 説明 | 優先度 |
|------|------|--------|
| MDファイル編集 | エディタでMarkdownを編集・保存 | 高 |
| 手動保存 | Cmd/Ctrl+S または保存ボタンで明示的に保存 | 高 |
| 未保存変更の警告 | 未保存時の画面遷移・ファイル切替時に確認ダイアログ | 高 |
| 競合検知 | 開いた時点の `updated_at` と保存時の最新値を比較し、差があれば警告 | 高 |
| 保存時の全文検索再インデックス | `content_text` を再抽出して更新 | 高 |
| リアルタイムプレビュー | 編集中にレンダリング結果をライブ表示 | 中 |
| 新規ファイル作成 | アプリ内で新しいMDファイルを作成 | 中 |
| 自動保存 | 一定間隔またはフォーカス離脱時に自動保存 | 低 |
| 編集履歴 | 変更履歴の保持・復元 | 低 |

### 技術候補

| ライブラリ | 特徴 |
|-----------|------|
| CodeMirror 6 | 軽量・高拡張性・モバイル対応良好 |
| Monaco Editor | VS Code同等の高機能・バンドルサイズ大 |

軽量さとモバイル対応を重視し、**CodeMirror 6** を第一候補とする。

### 画面イメージ

```
┌──────────────────────────────────────────┐
│ ヘッダー：ファイル名　　 保存　　閉じる  │
├─────────────────┬────────────────────────┤
│ エディタ        │ プレビュー             │
│                 │                        │
│ # 見出し        │ 見出し                 │
│ - リスト        │ • リスト               │
│ ```code```      │ code                   │
│                 │                        │

```

- PC: 左右分割（エディタ + プレビュー）
- スマホ: タブ切り替え（エディタ / プレビュー）

## 8. Excelセル編集機能要件

### 基本方針

xlsxファイルのセル値を直接編集・保存できる機能。業務用スプレッドシート相当の編集体験は目指さず、外出先での軽微な修正・メモ追記に最適化する。Markdown編集（Section 7）と同じUIパターン（Edit / View / Save ボタン、競合検知、全文検索の再インデックス）で統一する。

### スコープ

| 対応 | 不対応 |
|------|--------|
| セル値（文字列）の編集 | 行・列の追加/削除 |
| 既存の結合セル・列幅・書式・他シートの保持 | セル結合/解除の操作 |
| 全文検索インデックスの再生成 | セル書式（色・フォント・罫線）の編集 |
|  | 数式の編集（編集時に評価値で固定化） |

### 機能一覧

| 機能 | 説明 | 優先度 |
|------|------|--------|
| セル値編集 | 編集モードでセルをクリック→インライン入力→Enter/Blurでコミット、Escでキャンセル | 高 |
| 手動保存 | Saveボタンで明示的に保存（xlsxバイナリとしてStorageへupsert） | 高 |
| 未保存変更の警告 | View復帰時に確認ダイアログ、破棄時はStorageから再読み込み | 高 |
| 競合検知 | 開いた時点の `updated_at` と保存時の最新値を比較し、差があれば上書き確認 | 高 |
| 保存時の全文検索再インデックス | `content_text` を再抽出（全シートCSV化）して更新 | 高 |
| 結合セル・列幅の保持 | 編集対象セル以外のメタデータは変更しない | 高 |

### 保存フロー

```
編集モードでセルをクリック → インライン入力
        ↓ Enter / Blur で値をコミット
        ↓ Save ボタンで SheetJS の write() により xlsx バイナリを再生成
        ↓ Supabase Storage に upsert アップロード
        ↓ anyfolio_files.updated_at を更新
        ↓ content_text を再抽出して全文検索インデックスを更新
```

### 技術方針

- 編集用ライブラリは既存の **SheetJS (xlsx)** を流用（書き込みに `write(wb, { bookType: "xlsx" })` を使用）
- ワークブックを state として保持し、編集時は対象シートのみ shallow-copy して該当セルを差し替え。他シート・結合情報・列幅などは参照を保持
- 数式セル（`.f` を持つセル）を編集した場合は `{ t: "s", v: 新しい値 }` で完全置換し、数式を破棄
- `updateFileContent` は `string | Blob` を受け取れるよう拡張し、MD保存経路と共通化

### 注意事項

- 数式のあるセルを編集すると数式が失われ、評価値が固定値として書き込まれる
- セル書式の保持はSheetJS OSS版の `write()` が扱える範囲に依存
- 行/列の追加・セル結合の操作はスコープ外（必要な場合は元ソフト側で編集してから再アップロード）

## 9. ダウンロード機能要件

### 基本方針

Supabase Storage に保存されているファイルを、ユーザーがいつでも端末にダウンロードできる機能。閲覧中のファイルをワンタップで保存できることを主眼に置く。

変換を伴うアップロード（Section 3-5 / 3-6）が行われたファイルは、**元形式ではなく変換後のファイル**（pptx/ppt → pdf、txt/docx/doc → md）がダウンロードされる。元ファイルは保存されていないため、変換前形式への復元はスコープ外。

### スコープ

| 対応 | 不対応 |
|------|--------|
| 単一ファイルのダウンロード | 複数ファイルの一括ダウンロード（zip 化） |
| 閲覧中ファイルのビューアからの即ダウンロード | フォルダ単位のダウンロード |
| サイドバーのコンテキストメニューからのダウンロード | ダウンロード履歴・回数制限 |
| 変換後ファイル（pdf / md）のダウンロード | 元形式（pptx / docx / txt など）の復元ダウンロード |

### 機能一覧

| 機能 | 説明 | 優先度 |
|------|------|-------|
| ビューアからのダウンロード | ビューアヘッダーの Download ボタンで現在開いているファイルを保存 | 高 |
| サイドバーからのダウンロード | ファイル行のコンテキストメニューに「Download」項目を追加（既存の Rename / Move / Delete と並列） | 高 |
| ファイル名の保持 | 保存時のファイル名は `anyfolio_files.name`（変換後の拡張子）をそのまま使用 | 高 |
| 進捗・結果の通知 | アップロードと共通のステータスバッジで「Downloading... / Saved / Failed」を表示 | 中 |
| 一括ダウンロード | フォルダ単位または複数選択で zip アーカイブ化 | 低（Phase 2） |

### ダウンロードフロー

```
ユーザーが Download ボタン／メニュー項目をクリック
        ↓ Supabase Storage から Blob を取得
        ↓ Blob URL を <a download="filename"> に渡して保存
        ↓ ステータスバッジで成功／失敗を表示
```

### 技術方針

- Supabase JS SDK の `supabase.storage.from("anyfolio-files").download(path)` で Blob を取得
- `URL.createObjectURL(blob)` + `<a download>` で端末に保存し、使用後に `URL.revokeObjectURL` で解放
- 進捗・完了・エラーは Section 4-3 のアップロード機能と共通のステータスバッジで表示し、体験を統一
- RLS により、他ユーザーのファイルはダウンロード不可

### 注意事項

- 変換を伴うファイル（pptx/ppt/docx/doc/txt）は、元形式ではなく変換後の pdf / md としてダウンロードされる。この仕様はユーザーに明示する（ダウンロードメニューのラベル／ツールチップ等で示唆）
- iOS Safari では Blob ダウンロードの挙動が一部異なるため、必要に応じて署名付き URL (`createSignedUrl`) でのナビゲーションにフォールバック
- 大容量ファイル（例: 50MB 以上の PDF）はダウンロードに時間がかかるため、ステータスバッジでの進捗表示が重要
